# Changelog

User-facing changes to PerfLint for Unity. This project follows [Semantic Versioning](https://semver.org/).

## [Unreleased]

## [1.2.0] — 2026-07-16

The "one-click cleanup, verified" release — disable unused built-in modules and XR/AI-Navigation packages with a compile-verified auto-rollback safety net, catch GPU-Instancing flags that do nothing (or hurt), tighten WebGL/IL2CPP build size, and capture the shader variants your shipped players actually compile.

### Added
- **The scan panel now tells you which checks only see your open scene.** Most of PerfLint's rules scan the whole project, but a few — Static Batching memory (SBATCH001), GPU Instancing overlap (MAT004), Skinned Instancing (MAT005) and Mesh LOD — can only look at the scene(s) you currently have open. Scan with an empty or light scene and their results are silently partial (a real trap: scanning with no scene open makes the Static Batching memory bill read zero). A persistent note on the scan panel now names those checks and the scene(s) they're reading right now, and reminds you to open your heaviest scene and re-scan for the full picture. It updates live as you switch scenes — and if you open a different scene after a scan, it turns amber and prompts a re-scan, since the scene-level findings still on screen belong to the scene you scanned, not the one now open.
- **One-click disable for unused built-in modules (PKG001, Pro), with a compile-verified safety net.** The "unused built-in module" finding is no longer advice-only: a **Disable** button removes the module from `Packages/manifest.json` for you, then recompiles to verify. If any script still references the module and fails to compile, PerfLint automatically reverts the manifest — nothing is lost, and the finding comes back. Disabling is reversible either way (re-enable any time under Package Manager ▸ Built-in), which is why this one is safe to automate where irreversible actions aren't. It refuses to run while the project already has compile errors, so a pre-existing break is never mistaken for one the removal caused. Scanning and the finding stay free; the one-click execution is Pro.
- **PerfLint now tells you when the editor is running outdated code — its own, or your packages'.** When a project has C# compile errors, Unity cannot reload scripts — so a PerfLint update, or *any package change* (say, upgrading URP mid-migration), sits on disk while the whole session keeps running the pre-change assemblies, with no indication anywhere. The main window now detects exactly this (change newer than the running session + compilation failed) and shows a banner: fix the compile errors, or restart the editor. On healthy projects Unity reloads by itself, so the banner never appears there.
- **AI Migrate refuses to run on a stale session instead of wasting your credits.** In the state above, the API probes that guide a migration would describe the *old* loaded packages while compile verification judges against the *new* ones — every generated rewrite is doomed before it starts. AI Migrate (and its automatic retries) now detect this, stop, and tell you to restart the editor first, instead of burning generation rounds on a rigged game.
- **AI Migrate for `GetInstanceID` (MIG.GetInstanceID, Pro).** Newer Unity 6 releases deprecate `Object.GetInstanceID()`, and the correct fix is not a rename — the `EntityId`→`int` implicit conversion is deprecated there too, so the id's receivers have to migrate with the call. The Migration Assistant now rewrites the whole file, deciding per call site between the two sound strategies: where the id is only a unique key (dictionary keys, job handles — especially when it flows into other files' `int` signatures), it switches to a local per-instance counter and touches no types at all; where real engine-object identity is required, it moves to `GetEntityId()` and propagates the `EntityId` type through the file's receivers. Before anything is sent, PerfLint asks your installed engine (by reflection) whether `GetEntityId()` exists and how deprecated the implicit conversion is, and hands the model those facts — this API surface is newer than any model's training data, so it's never left to guess. Same contract as every AI Migrate: full-file disclosure before sending, compile verification, automatic rollback, error-driven retries.
- **New check: GPU Instancing that overlaps Static Batching (MAT004).** Enabling "GPU Instancing" on every material — a common bulk "optimization," and one AI assistants like to apply wholesale — is inert on any object that's marked Batching Static: static batching takes priority, so those renderers are drawn from the build-time Combined Mesh and the Instancing flag is never used. Worse, under URP/HDRP that flag also keeps the material out of the SRP Batcher for its non-static users. PerfLint now scans your loaded scene(s) and, when Static Batching is enabled for the platform, reports how many static-batched renderers place a GPU-Instancing material, names the top materials by overlap, and points you at the objects. It's an Info-level review prompt with no auto-fix on purpose — the same material may be instanced legitimately on non-static or runtime-spawned copies (grass, projectiles, pooled objects), so the call is yours per material. Complements MAT002, which flags instancing at the material-asset level.
- **New check: GPU Instancing on skinned-only materials (MAT005).** The other half of the "tick GPU Instancing on everything" trap: skinned meshes are *never* drawn through the instancing path — each Skinned Mesh Renderer is deformed and drawn on its own, no matter how many identical copies you have — so the flag does nothing on a material only characters/creatures use, and under URP/HDRP it's a small net loss (it drops the material from the SRP Batcher). PerfLint flags materials that, across your loaded scene(s), are used only by Skinned Mesh Renderers and nowhere by a Mesh Renderer, and lists them so you can clear the flag. Info-level, no auto-fix — if the same material is also on non-skinned meshes elsewhere (other scenes, prefabs, runtime-spawned), instancing can still help there, so it's a per-material review.
- **Capture shader variants from a real build — and stream them back into the editor.** What the editor records in Play Mode is what the *editor* renders; your built game compiles a different variant set per platform, graphics API and quality level. The PerfLint Shaders panel now has a device-capture workflow: flip one toggle (it turns on Graphics Settings → Log Shader Compilation), make a Development Build, and play it. Desktop players (Windows/macOS/Linux) stream every variant they compile back into the panel live while attached (Build & Run attaches automatically, the same channel the Profiler uses) and also keep a local capture file you can import later; for Android/iOS, dump the device log (adb logcat, or the Xcode console) and import it — the panel reads raw player logs directly. Merge the capture into a `.shadervariants` collection (Pro): every record is validated against the shader it names (nothing is ever guessed in), shaders not in your project are reported honestly, and the merged collection is wired straight into startup warm-up — so the variants your players really use are precompiled on the devices that matter, and first-use hitches disappear. Release builds are untouched — the recorder isn't even compiled into them — and capture never leaves your machine or LAN: nothing is uploaded, ever.
- **New check: WebGL build compression is disabled (PROJ010).** If Publishing Settings ▸ Compression Format is set to Disabled, your .wasm and .data files ship uncompressed — downloads are commonly 2–4× larger than with Brotli, and on web game platforms download size *is* load time. The finding explains the tradeoffs (Brotli is ~20% smaller than Gzip and works over HTTPS, which itch.io, Poki, CrazyGames and similar platforms already serve; self-hosters without Content-Encoding headers should also enable Decompression Fallback) and offers a one-click switch to Brotli (Pro). Deliberately quiet if you chose Gzip — that's a legitimate self-hosting tradeoff, not a mistake. Only reported when your active build target is WebGL.
- **New check: Strip Engine Code is off (PROJ011).** On IL2CPP platforms (including WebGL), Strip Engine Code removes unused native engine code from the binary — often several MB — and is a separate switch from Managed Stripping Level, which only covers C# assemblies. PerfLint now flags it when it's off and the setting would actually apply (IL2CPP backend or WebGL target), with a one-click enable (Pro). The guidance covers the one real caveat: engine types referenced only by runtime-loaded content (AssetBundles/Addressables) must be preserved via link.xml and tested in a real build.
- **New check: IL2CPP Code Generation favors speed on a size-sensitive target (PROJ012).** On WebGL and IL2CPP mobile targets, the "Faster runtime" code generation mode expands every generic instantiation, inflating binary size and build time; "Faster (smaller) builds" shares generic method bodies at a small runtime cost that rarely shows outside generics-heavy hot loops. PerfLint suggests the size-oriented mode where download size matters most, with a one-click switch (Pro) whose confirmation states the tradeoff honestly — profile afterward if your per-frame code leans on generics.
- **One-click disable now covers the unused-XR and unused-AI-Navigation findings too (PKG002/PKG003, Pro).** The same compile-verified safety net that backs the unused-module disable: PerfLint edits the manifest, recompiles to verify, and auto-reverts if anything still references what was removed. The XR finding removes its whole module group (vr/ar/xr) in a single edit with a single verification — the modules depend on each other, so removing them one at a time would be blocked by Unity's own dependency rules. The AI Navigation finding removes the registry package; re-add it any time from the Unity Registry. Both remain fully reversible, and scanning stays free.
- **Re-analyze Addressables duplicates without a full scan.** The Addressables duplicate rules (AADUP001 / AARES001) now have a **Re-analyze** button in their group header. After you manually move an asset out of a Resources folder — the refactor AARES001 recommends, which the tool can't safely do for you — click it to re-run just the Addressables duplicate analysis (a few seconds) instead of a full project scan (which can be ~100s on large projects). The AARES001 entry clears and the asset reappears under AADUP001 as an ordinary extractable duplicate, ready for one-click extraction. Scanning stays free.

### Changed
- **The Static Batching finding (SBATCH001) now names the build-size cost too — including one Unity's own Build Report hides.** Static Batching's build-time Combined Meshes don't only sit in runtime memory; they're baked into the player, so disabling it also shrinks the build (a real 3D Game Kit case dropped the Complete build size 2.1 GB → 1.5 GB). The catch worth knowing: Unity's Build Report itemises "usage by category" for *user-generated assets only*, so the combined geometry appears under no category — not even Meshes — and surfaces only in the "Complete build size" total. The finding now spells this out, so you can weigh the trade-off on both memory and download size.
- **Mesh compression advice now respects lightmap UVs.** The "Mesh compression is off" suggestion (PERF.MSH002) no longer offers a one-click fix on models that generate Lightmap UVs. Mesh compression quantizes *every* vertex channel — including the lightmap UV (UV2) — so enabling it on a lightmapped model can bake visible seams and cracks into baked surfaces. That's a trade-off you have to eyeball, not a safe blanket win: the finding still appears (the build-size cost is real) but routes you to set the level manually and inspect the lightmapped result, so a bulk "fix all" can never silently degrade a lightmapped scene. Models without lightmap UVs keep the one-click fix.

### Fixed
- **Assets used only through Project Settings are no longer wrongly flagged as unreferenced (ASSET.UNREF001).** The unreferenced-asset check walked build scenes and Resources, but not the assets your project references from **Project Settings** — most visibly the Input System's project-wide actions asset (`InputSystem_Actions.inputactions`), which ships in the build via Preloaded Assets and so appeared as "dead weight" on every Unity 6 / URP-template project. PerfLint now also treats assets referenced from `ProjectSettings/` (preloaded assets, always-included shaders, the active render-pipeline asset, project-wide input actions, and the like) as reachable, so these genuine references stop showing up as false positives.
- **Modules your project can never actually disable are no longer reported as "unused."** Unity keeps a built-in module installed as long as any installed package depends on it — Scriptable Render Pipeline Core, for example, depends on the Terrain module, so no URP/HDRP project can ever disable Terrain. PerfLint now checks that dependency graph during the scan and simply doesn't report such modules: advice you cannot execute is noise. Multi-level cleanups now proceed naturally leaf-first — disable what's offered, rescan, and the next layer (if it's genuinely free) appears; no dead-end dialogs. The XR-module finding applies the same logic to its group: only the modules that can really leave together are offered.
- **Disabling a built-in module that another package needs is now refused upfront — not silently ignored.** Unity keeps a module installed as long as any installed package depends on it (Tilemap requires Physics 2D, for example), so removing its manifest line did nothing: Unity resolved the module right back in, while the finding disappeared as if the disable had worked. PerfLint now checks the same dependency information Package Manager uses and tells you plainly: "required by X — disable X first." One-at-a-time module disabling also no longer gets stuck for the whole session when a disable triggers no recompile: a verification that can no longer receive a verdict clears itself after a few minutes instead of blocking every further disable until you restart the editor. When a rule has exactly one actionable finding, the rule-level batch button no longer appears — the per-row button right below it already does the same thing. Project-wide settings checks (which can only ever have one finding) were showing it on every scan.
- **Script analysis no longer turns itself off when you switch build targets.** Unity stores scripting defines per platform, and enabling deep script analysis used to write its define only for the platform active at the time — so switching (say, Windows → WebGL) brought the "not enabled" banner back even though everything was installed. Enabling now applies to all build targets at once, and projects enabled by earlier versions heal automatically on load: if the module's files are present and it was enabled for another target, the current target is fixed up with a one-time recompile (a console message notes when this happens). Projects where you removed the module's DLLs are left alone. Compile verification now only trusts a compilation pass that started *after* the fix was written. Previously, an automatic retry written while an earlier compilation was still in flight — easiest to hit on projects that don't compile before the migration, which is exactly the upgrade scenario — could be rolled back on the strength of that older pass's errors, so a perfectly good rewrite never got its own compile. Such writes now simply stay pending and are verified by the next compilation of the real content. Automatic retries also restate the URP capability probe results to the AI each round, keeping regenerated migrations on the pass shape your URP version actually exposes.
- **Scanning very large projects no longer risks an out-of-memory editor crash.** Some checks load assets to inspect them — the texture-compression check (PERF.TEX005), the material and audio-import scans — and on projects with many thousands of assets those loads accumulated in graphics/native memory across the scan until the editor could run out and crash. The scan now reclaims that memory periodically, so peak usage stays flat regardless of project size — with no change to what it reports and no measurable slowdown. A memory watchdog also backs this up: if any scan ever approaches the graphics-memory ceiling, it reclaims automatically and logs a warning, so a heavy scan can no longer take the editor down.

## [1.1.0] — 2026-07-04

The "stuck Unity upgrade" release — turns a project that won't compile after a Unity 6 upgrade into a fixable checklist, and fixes most of it for you.

### Added
- **AI Migrate (Pro): one-click structural migrations, starting with RenderTargetHandle → RTHandle.** Some migrations can't be fixed by replacing one line — URP's removed `RenderTargetHandle` changes declarations, overrides, allocation and release across the whole file. AI Migrate rewrites the entire file following a curated migration playbook, probes your actual URP version by reflection to pick the right pass shape (compatibility mode vs. Render Graph — the model follows facts, not guesses), shows you exactly what changed (unchanged sections are folded), and only writes after you approve. If the rewrite doesn't compile, PerfLint feeds the compiler errors back to the AI and retries automatically (up to 2 rounds per click). Safety first: every round is compile-verified and **auto-rolls back on failure — your project is never left worse than before the click**. Privacy: unlike AI Fix (snippet-only), this sends the whole file to the AI provider — the panel states the exact line count and provider before every send, and nothing is sent until you click Generate. Current limits, stated honestly: files up to 500 lines, and you should still eyeball the visual result in your scene — a compile pass can't judge rendering. (This release ships several migration recipes — see the next entries.)
- **Any compile error is now a finding — and most are fixable.** PerfLint captures your project's C# compiler errors as they happen and turns them into per-file findings (Migration domain, Critical), so a project that won't compile after a Unity upgrade shows you exactly which scripts are blocked and why — even errors no curated rule knows about. Errors in package code get a separate summary pointing at the usual root cause (package version mismatch). Each broken script offers **AI Migrate (Pro)**: a whole-file rewrite guided by that file's exact error list plus a playbook of common upgrade breakages (FindObjectOfType family, WWW→UnityWebRequest, GetInstanceID→GetEntityId including the EntityId type ripple, Application.LoadLevel, and more), with the same compile-verify → auto-rollback → error-driven retry loop as all migrations. Note: fixing one file can surface follow-up errors in files that call into it — rescan after each fix; PerfLint reports these honestly rather than chaining rewrites automatically.
- **AI Migrate now repairs broken shaders too (Pro)**: on a "Shader fails to compile" finding (SHDR004), the Migration Assistant rewrites the file the compiler error actually points at — usually an included .hlsl, not the .shader shell — guided by your project's real URP version, the ShaderLibrary includes that actually exist on disk, and the shader's full error list. Verification is immediate and thorough: the shader is re-imported and its passes are actively re-compiled (catching lazy HLSL errors a plain re-import would miss); a failed rewrite rolls back automatically and feeds the fresh compiler errors back to the model (up to 2 retries per click). Hard guards refuse any rewrite that renames the shader or drops a material property — materials bind by those names. Same privacy contract as all AI Migrate actions: the panel states the exact file and line count before anything is sent.
- **"Why is everything magenta?" now has full coverage.** Two new checks join the existing pipeline-mismatch rule (MAT001), so each of the three common causes of pink materials gets its own precise diagnosis:
  - **Shader fails to compile (SHDR004)**: shaders whose source no longer compiles — typical after a Unity or render-pipeline upgrade breaks include paths, macros or syntax. The finding quotes the first compiler error (with line number) and the error count, and points you at the migration path. Reported from Unity's own recorded compile state, so it's deterministic.
  - **Material has a missing shader (MAT003)**: materials whose shader reference is None or points to a deleted/uninstalled shader. Previously these were silently skipped by the scan.
- **Migration check for `GetInstanceID` (MIG.GetInstanceID)**: newer Unity 6 releases deprecate `Object.GetInstanceID()` in favor of `GetEntityId()` — at error level it stops your project from compiling. PerfLint detects this precisely by asking your installed engine (reflection over the actual `[Obsolete]` attribute), so there are no false alarms on editors where the API is still current. Heads-up in the guidance: this is not a plain rename — the `EntityId`→`int` implicit conversion is obsolete there too, so the variables and dictionary keys receiving the id must change type along with it.
- **Migration check for URP's `RenderTargetHandle` (MIG.RenderTargetHandle)**: deprecated since Unity 2022.1 (URP 13) and removed in Unity 6, this type is one of the most common compile blockers when bringing older URP content (store assets, samples, custom render passes) to Unity 6. PerfLint now pinpoints every usage with line-level Locate and lays out the RTHandle migration path (allocate in `OnCameraSetup` via `RTHandles.Alloc` / `RenderingUtils.ReAllocateIfNeeded`, pass handles directly, release explicitly). Reported on Unity 2022.1+ only — on 2021 LTS the API is still current, so no noise there.
- **Mipmap Streaming advisor**: when a sizable pool of your scene textures could be served on demand, PerfLint quantifies it ("~X MB of textures could load only the mip levels the camera needs") and can enable Texture Streaming project-wide in one click — Quality Settings switch plus the per-texture Stream Mip Maps flags (Pro). Also flags the half-dead configuration where textures request streaming but the global switch is off.
- **Texture Streaming tuning deck** in the Runtime Profiler: live readouts (resident vs. desired vs. budget-capped texture memory, "streaming is currently saving ~X MB"), an over-budget warning when the Memory Budget is visibly costing quality, and all streaming parameters editable live in Play Mode — the mip debug view SRP doesn't ship. Unity reverts Play Mode changes to Quality Settings on exit, so the deck remembers what you tuned and offers a one-click apply back in edit mode (edit-mode changes persist directly). Every parameter belongs to the active quality level, which the deck displays so a runtime level switch never looks like lost tuning.
- **Static Batching memory bill (PERF.SBATCH001)**: static batching copies every batched instance's vertices into build-time Combined Meshes. PerfLint now estimates that hidden cost for your loaded scenes, names the top contributing meshes, and offers a one-click toggle-off (Pro) — presented as an informed frame-time ↔ memory trade-off, not a blind recommendation.
- **AssetBundle built-in duplication (ASSET.ABDUP002)**: built-in resources (default particle texture, built-in shaders, default font…) referenced by multiple bundles get silently copied into each — now detected with example referencing assets and a concrete replacement recipe.
- **Addressables compression check (ASSET.AACOMP001)**: local groups built with LZMA (slow full-decompress loads with zero download benefit) are flagged, with a one-click switch to LZ4 (Pro).
- **Bundle granularity check (ASSET.AAGRAN001)**: judged from your last real Addressables build output — flags fragmentation (many sub-512KB bundles: header overhead, OS file-handle limits) and oversized bundles (>50MB), with the 2–5MB mobile reference band.
- **Resources ↔ Addressables duplication (ASSET.AARES001)**: assets living in Resources folders that Addressables content also bakes copies of — the player always ships the Resources copy, so every dependent bundle adds an extra one. This is where the biggest offenders hide (a real project had one TMP font atlas duplicated 74× ≈ 1.16 GB of build size), and they're invisible to bundle-duplicate analysis because Resources assets aren't valid Addressable candidates. Report-only with concrete guidance (move out of Resources; TMP-specific notes when applicable).

### Changed
- **AI Migrate now spells out its credit cost.** On the hosted AI service, each request costs one credit — and because a whole-file migration can automatically retry after a failed compile, it may use a few. The panel now says so up front. (Bring-your-own-key users pay their own tokens and are never counted.)
- **Critical severity now means "this blocks your project."** Findings that stop compilation or guarantee broken rendering are now Critical (red) instead of Warning: removed APIs (`Application.LoadLevel`, `GUIText`/`GUITexture`, legacy particles), `RenderTargetHandle` on Unity 2023.1+/6 (error-level there; still Warning on 2022 where it merely warns), error-level `GetInstanceID`, and shaders that fail to compile. Deprecation warnings that don't block anything stay Warning — so a red badge in your scan now reliably means "fix this to get the project running." Unresolved assembly-definition references stay Warning accordingly: Unity skips them and compiles anyway (the Inspector greys them out), so they're usually migration leftovers to clean up — the finding now explains exactly when they do and don't matter.
- The shareable **HTML report** now carries a small, unobtrusive "Generated by PerfLint for Unity — perflint.dev" line in its footer. It's a plain link that only opens when clicked — the report still loads nothing and calls out to nothing on its own, and your scan is never uploaded.

### Fixed
- **Fixed a compile error on Unity 2021.3** introduced with the Texture Streaming tuning deck: two UI field types moved namespaces in Unity 2022.1, so the deck failed to compile on 2021 LTS. All supported versions (2021.3 through Unity 6) now compile and pass the full test suite.
- **Duplicate-packing findings are no longer hidden by your ignore list.** Ignored paths (Plugins/, Dependencies/…) are meant to silence import-settings advice on third-party assets — but a third-party asset duplicated into many bundles bloats *your* build, and fixing it never edits the third-party asset. Duplication findings (Addressables, AssetBundles, Resources overlap) now bypass ignore patterns; everything else still respects them. In a real project this was hiding the single biggest offender (a TMP font duplicated 74×).
- **Addressables duplicate detection (ASSET.AADUP001) no longer misses same-group duplication** — the biggest fish were slipping through. An asset baked into many bundles of ONE group (typical with one-bundle-per-scene groups; a real project had a font duplicated 74× ≈ 1.16 GB) previously wasn't reported at all, because duplicates were counted per group. Duplicates are now counted per bundle, matching the official Build Report. Findings also explain two special cases when they apply: same-group duplication (extraction still fixes it) and assets living in Resources folders (the Resources copy always ships — moving the asset out of Resources is the complete fix, typical for TMP fonts).
- **Finding details are now visible in the main panel.** Every finding row with details gets a ▸ caret — click the row to expand the full explanation (counts, guidance, trade-offs). Project-wide findings (no asset path) start expanded, since their substance lives entirely in the detail. Previously details only reached CSV/HTML exports and AI Explain, which left detail-heavy findings looking empty.
- **"Extract to shared group" (Addressables dedup) batch** now tells you what actually happened. After a rule-level "Extract all", a summary dialog reports how many assets were extracted, how many were skipped (read-only package assets can't be made Addressable), and — most importantly — the official duplicate-dependency count **before vs. after**, so you can see whether it actually reduced duplication. When it doesn't (common when the duplication is dominated by scene bundles that bake their own copies), it says so and points to the real fix instead of silently doing nothing. The batch also runs far faster (a single save instead of one per asset) with a progress bar, and no longer looks like "nothing happened" on success.
- **Runtime stutter spike (FPS003)** attribution is now reliable. The spike frame's call stack is captured the moment the freeze happens, so the finding pinpoints the script that triggered it (with Locate) instead of occasionally attributing the spike to an unrelated, quiet frame. When a spike occurs outside the sampling window and can't be captured, the finding now says so and points you to the Unity Profiler, rather than showing a misleading cause.

## [1.0.0] — 2026-06-30

First stable release — paid subscriptions are now live. What's included:

- **Local scan** across Performance, Assets, Migration, and Project Settings — one click, runs entirely on your machine.
- **Mobile-focused checks**: textures whose compression silently fell back to uncompressed, Multithreaded Rendering, Optimize Mesh Data, stereo clips that could ship as mono, and oversized fonts — each with a one-click fix where it's safe.
- **Project health score (0–100)** and a self-contained, shareable **HTML report**.
- **One-click & batch fixes** for safe, deterministic issues (Pro).
- **Merge duplicate assets** (Pro): keep one copy, redirect every reference across the project, then delete the rest — one click. Pick which copy to keep (defaults to the most-referenced one); guarded and previewed before it runs; commit to version control first.
- **Shader variants**: see how many variants each shader can produce, then **record** the ones your project actually uses and **warm them up** at startup (Pro) so they don't hitch on first use.
- **AI Fix & Explain** — zero-config out of the box, or bring your own API key. Every AI change is shown as a diff for review and is compile-verified with automatic rollback.
- **Local-first, zero telemetry** — your code and art assets are never uploaded.
